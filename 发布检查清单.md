# 鲁班AI 正式发布检查清单

基于代码审查得出的分级优化建议。按优先级分 P0（必须修复）、P1（强烈建议）、P2（值得做）、P3（远期）。

---

## P0 — 必须修复（安全与稳定性）

### 1. XSS 防护

**现状**: `dom.js` 的 `h()` 函数在创建元素时直接设置 `textContent`，但 `innerHTML` 在 `Markdown.js` 等处使用。所有用户输入的字段值可能含 HTML 标签。

**建议**:
```js
// Markdown.js 及所有 innerHTML 使用点
element.textContent = userInput;  // 而不是 innerHTML
// 如果确实需要渲染富文本，用 DOMPurify 或类似库消毒
```

**风险**: 用户在字段中输入 `<img onerror="alert(1)">` 会在其他用户的浏览器中执行任意脚本。

### 2. 请求体大小限制加固

**现状**: `_helpers.js` 的 `collect()` 有 2MB JSON 限制和 20MB 文件限制，但大型 CSV/XLSX 导入时内存直接缓冲。

**建议**: 
- CSV 导入改用流式处理而非全量缓冲
- 增加 `/api/apps/import` 的限制检查

### 3. SQLite 写并发

**现状**: 单 `DatabaseSync` 实例，不支持并发写入。生产环境中如果部署多个进程（如 Render 的多实例），每个进程会创建独立的 SQLite 连接，导致数据不一致。

**建议**:
- 短期：确保只运行单进程
- 长期：迁移到 `better-sqlite3` 的 WAL 模式 + `busy_timeout`，或引入 SQLite 代理层

### 4. 路径穿越加固

**现状**: `isInside()` 函数已有基本防护，但 `serveStatic` 使用 `resolve` + `.` 前缀，需要确认在 Windows 和容器环境下的表现。

**建议**: 增加 `decodeURIComponent` 防护，防止双重编码绕过。

## P1 — 强烈建议

### 1. 错误处理统一化

**现状**: 错误通过 `throw notFound('消息')` 抛出，在 `server.js` 的 catch 块中统一捕获。但 `notFound()` 返回的是普通 Error 加 `.status` 属性，没有标准 Error 类。

**建议**:
```js
class HttpError extends Error {
  constructor(status, message, details) {
    super(message);
    this.status = status;
    this.details = details;
  }
}
// 统一用 throw new HttpError(404, '消息')
```

### 2. AI 会话确认机制持久化

**现状**: `PENDING_CONFIRMS` 存在内存 Map 中。服务器重启后所有待确认的操作丢失。

**建议**: 将待确认操作写入 `ai_execution_logs` 表，加上 `status = 'pending_confirmation'`，启动时恢复。

### 3. 添加 CSP 和 Security Headers

**建议**:
```js
// server.js
res.setHeader('content-security-policy', "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'");
res.setHeader('x-content-type-options', 'nosniff');
res.setHeader('x-frame-options', 'DENY');
```
（注意：当前使用内联 onclick 和 style，CSP 需配合 `unsafe-inline` 或改为事件委托）

### 4. 数据库迁移机制

**现状**: `migrate()` 用 `CREATE TABLE IF NOT EXISTS` 创建表。版本升级无法修改已有表结构。

**建议**:
```js
const SCHEMA_VERSION = 1;
function migrate(db) {
  const version = db.prepare('PRAGMA user_version').get()?.user_version || 0;
  if (version < 1) { /* v1 migrations */ }
  if (version < 2) { /* v2 migrations */ }
  db.exec(`PRAGMA user_version = ${SCHEMA_VERSION}`);
}
```

### 5. 请求日志与监控

**建议**:
```js
// 添加结构化 request logging
const start = Date.now();
// ... handle request ...
const duration = Date.now() - start;
console.log(JSON.stringify({
  method, pathname, status, duration, userAgent: req.headers['user-agent']
}));
```

### 6. Playwright E2E 测试

**现状**: Playwright 在 devDependencies 中但未被使用。关键用户流程（创建应用、编辑记录、四象限视图操作、字段设置）缺少浏览器级测试。

**建议**: 补充至少 3 个 Playwright 测试：
- 创建应用 → 添加记录 → 切换视图 → 验证字段显示
- 四象限视图字段设置 → 隐藏字段 → 验证隐藏
- 甘特视图字段设置 → 调整可见字段

## P2 — 值得做

### 1. 速率限制

**建议**:
```js
// 简单的内存限流
const rateLimits = new Map();
function rateLimit(key, maxPerMinute = 60) {
  // 基于 IP 或 sessionId 做限流
}
```

### 2. 前端加载状态

**现状**: 数据加载时没有骨架屏或 loading indicator。慢网络下用户看到白屏。

**建议**: `app.js` 的 `loadApp()` 等函数中增加加载状态，渲染时显示简化的 loading 骨架。

### 3. 表单验证前端化

**现状**: 必填字段校验在后端，前端提交后才报错。

**建议**: `CellEditor.js` 的 `inputForField()` 中根据 `field.required` 添加前端 required 属性，在 `RecordModal.js` 保存前做前端校验。

### 4. 数据导出编码

**现状**: CSV 导出用 `toCsv()`，但未做字段值内的逗号/引号转义。

**建议**: 检查 `export.js` 的 CSV 序列化是否处理了字段值内包含逗号、换行、引号的情况。

### 5. 路由声明式重构

**现状**: `server.js` + `runtime.js` 的路由分发靠手写 if/else 条件链。新增 API 容易遗漏。

**建议**: 用简单的路由表：
```js
const routes = [
  ['GET', '/api/apps', handleAppApi],
  ['POST', '/api/apps/generate', handleGenerateApp],
  // ...
];
```

### 6. 环境变量配置化

**建议**: 除了 PORT，将以下配置提取到环境变量：
- `UPLOAD_DIR` — 上传目录路径
- `DATA_DIR` — 数据库目录路径
- `LOG_LEVEL` — 日志级别
- `CORS_ORIGIN` — 跨域允许的来源
- `MAX_UPLOAD_SIZE` — 上传大小限制

### 7. SSE 连接清理

**现状**: `routes/ai.js` 的 `keepalive` 定时器在异常退出路径有泄漏风险。

**建议**: 使用 `res.on('close', () => clearInterval(keepalive))` 确保清理。

## P3 — 远期规划

### 1. 多租户与认证

**现状**: 完全无用户系统。同一部署的所有用户共享数据。

**建议**: 增加 `users` 表，`apps` 表加 `userId` 外键，增加登录/注册页面。

### 2. 数据库备份

**建议**: 提供 `npm run backup` 命令，在数据目录创建 SQLite 的 `.backup`。

### 3. Docker 化

```dockerfile
FROM node:25-alpine
WORKDIR /app
COPY . .
RUN npm install --production
EXPOSE 5173
CMD ["node", "src/server.js"]
```

### 4. CI/CD 管线

**建议**: `.github/workflows/test.yml` — 每次 PR 自动跑 `npm test`，包含 Node 25 环境。

### 5. 性能基准

**建议**: 补充测试验证 10000 条记录下的列表页渲染时间和 API 响应时间。

### 6. 国际化

**现状**: 所有 UI 文本和错误消息为中文。国际用户需建立 i18n 机制。

## 已有但容易被忽略的优势

检查过程中发现以下做得好的地方，值得保留：

| 措施 | 位置 |
|------|------|
| SQLite WAL 模式 | `db.js:16` — 写入性能好 |
| 外键约束 | `db.js:17` — 数据完整性 |
| 路径穿越防护 | `_helpers.js:170-172` — `isInside()` |
| 上传文件名消毒 | `_helpers.js:120-122` — `safeOriginalName()` |
| MIME 类型白名单 | `_helpers.js:125-134` — `safeExtension()` |
| 请求大小限制 | `_helpers.js:54-70` — `collect()` 2MB/20MB |
| 公式引擎安全 | `formula.js:206` — 禁止 `globalThis` |
| 包协议校验 | `packageProtocol.js:330-421` — 多维度校验 |
| 测试隔离 | 每个 HTTP 测试用独立 SQLite 文件 |
